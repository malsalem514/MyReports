## Step-by-Step: Register a New App for SSO

### 1. Create the App Registration

1. Go to **Azure Portal** → search **"App registrations"**
2. Click **"+ New registration"**
3. Fill in:
   - **Name**: A descriptive name (e.g., "MyReports (HR dashboard)")
   - **Supported account types**: "Accounts in this organizational directory only" (single tenant) unless external users need access
   - **Redirect URI**: Leave blank for now (we'll add it next)
4. Click **Register**
5. Note the **Application (client) ID** — the app will need this

### 2. Add the Redirect URI

This is the most critical step and the #1 source of errors.

1. In the app registration, click **"Authentication"** in the left menu
2. Under **"Platform configurations"**, click **"+ Add a platform"**
3. Select **"Web"** (NOT "Single-page application", NOT "Mobile and desktop")
4. Enter the **exact** redirect URI for your application (see framework-specific formats below)
5. Click **"Configure"**
6. Click **"Save"** at the top of the page

### 3. Create a Client Secret

1. Click **"Certificates & secrets"** in the left menu
2. Click **"+ New client secret"**
3. Add a description (e.g., "Production") and choose an expiry
4. Click **"Add"**
5. **Immediately copy the "Value"** (not the "Secret ID") — it will only be shown once
6. Give the Value to the application developer for their environment config

### 4. Configure API Permissions (if needed)

Most SSO apps only need the default `User.Read` permission. If the app needs additional Microsoft Graph permissions:

1. Click **"API permissions"** in the left menu
2. Click **"+ Add a permission"** → **"Microsoft Graph"** → **"Delegated permissions"**
3. Add needed permissions (e.g., `openid`, `profile`, `email`, `User.Read`)
4. If admin consent is required, click **"Grant admin consent for Jesta I.S."**

## Redirect URI Formats by Framework

Each web framework uses a different callback URL pattern. The redirect URI registered in Azure **must exactly match** what the application sends.

### NextAuth v5 (Auth.js) — used by MyReports

```
https://<domain>/api/auth/callback/microsoft-entra-id
```

The path `/api/auth/callback/microsoft-entra-id` is auto-generated by NextAuth based on the provider ID. Do not change this unless the developer has customized the provider ID.

### ERPNext / Frappe — used by MyProjects

```
https://<domain>/api/method/frappe.integrations.oauth2_logins.login_via_office365
```

This is Frappe's built-in Office 365 OAuth handler. It is always this exact path.

### Other Common Frameworks

| Framework | Redirect URI Pattern |
|-----------|---------------------|
| NextAuth v5 (Auth.js) | `https://<domain>/api/auth/callback/microsoft-entra-id` |
| ERPNext / Frappe | `https://<domain>/api/method/frappe.integrations.oauth2_logins.login_via_office365` |
| Laravel Socialite | `https://<domain>/auth/callback` |
| Django (allauth) | `https://<domain>/accounts/microsoft/login/callback/` |
| Spring Boot | `https://<domain>/login/oauth2/code/azure` |
| Express (Passport) | `https://<domain>/auth/openid/return` |

Always confirm the exact path with the developer before registering.

## Redirect URI Rules

These rules are enforced by Azure AD and cannot be bypassed:

1. **Exact match required** — the URI must match character-for-character, including:
   - Protocol (`https://` vs `http://`)
   - Domain name (including subdomains)
   - Port number (if non-standard; port 443 is implied for `https://`)
   - Full path (including trailing slash vs no trailing slash)
2. **Web platform for server-side apps** — apps that exchange codes on the server (most web apps) must use "Web", not "SPA"
3. **SPA platform for browser-only apps** — only use "SPA" for pure JavaScript/React/Vue apps with no server-side code exchange
4. **No wildcards** — Azure AD does not support wildcard redirect URIs
5. **`https://` required** — except for `http://localhost` during development
6. **Copy-paste, don't type** — always copy the URI from the developer; never type it manually

## Common Errors and Fixes

### AADSTS50011 — Redirect URI Mismatch

```
The redirect URI '...' specified in the request does not match the redirect URIs configured for the application
```

**Cause**: The redirect URI sent by the app doesn't match what's registered in Azure AD.

**Fix**:
1. Use `az ad app show` (see above) to check what's actually registered
2. Compare character-by-character with what the app sends
3. Common mistakes:
   - Wrong domain (e.g., `myprojects` vs `myreports`)
   - Old hostname still registered (e.g., `srv-test-Docker-Musa:3000` instead of `myreports.jestais.com`)
   - Port number included when it shouldn't be (port 443 is implicit for HTTPS)
   - Trailing slash difference (`/callback` vs `/callback/`)
   - Wrong path (using one app's callback path for another app)

### AADSTS900561 — POST Not Accepted

```
The endpoint only accepts POST requests
```

**Cause**: The redirect URI is registered under "SPA" platform instead of "Web".

**Fix**: In the app's Authentication page, delete the URI from the SPA section and re-add it under the "Web" platform. Server-side apps (NextAuth, ERPNext, Django, etc.) require the "Web" platform because they exchange the authorization code on the server using a client secret.

### AADSTS50020 — User Not in Tenant

```
User account 'user@domain.com' from identity provider 'live.com' does not exist in tenant
```

**Cause**: The user has a personal Microsoft account (live.com/outlook.com) instead of a work account in the Jesta I.S. tenant, or the user is an external/guest user who hasn't been invited.

**Fix** (choose one):
- **Invite the user as a guest**: Azure Portal → Azure Active Directory → Users → "+ Invite external user"
- **Enable multi-tenant**: Change "Supported account types" to "Accounts in any organizational directory and personal Microsoft accounts" (less secure)

### InvalidCheck: pkceCodeVerifier — PKCE Cookie Issue

```
pkceCodeVerifier value could not be parsed
```

**Cause**: The app is behind a reverse proxy (e.g., nginx) and the PKCE verification cookie isn't being set correctly. This happens when `__Secure-` prefixed cookies require the `Secure` flag but the proxy doesn't pass the right headers.

**Fix** (for NextAuth v5):
1. Ensure the reverse proxy sets `X-Forwarded-Proto: https`
2. Configure the PKCE cookie explicitly in `auth.ts`:
   ```typescript
   const useSecureCookies = process.env.NEXTAUTH_URL?.startsWith('https://') ?? false;
   const cookiePrefix = useSecureCookies ? '__Secure-' : '';

   export const { handlers, auth, signIn, signOut } = NextAuth({
     trustHost: true,
     cookies: {
       pkceCodeVerifier: {
         name: `${cookiePrefix}authjs.pkce.code_verifier`,
         options: {
           httpOnly: true,
           sameSite: 'lax',
           path: '/',
           secure: useSecureCookies,
         },
       },
     },
     // ... providers, callbacks, etc.
   });
   ```

### 502 Bad Gateway (nginx)

**Cause**: Azure AD auth cookies are very large and exceed nginx's default buffer sizes.

**Fix**: Increase proxy buffer sizes in nginx config:
```nginx
proxy_buffer_size        128k;
proxy_buffers          4 256k;
proxy_busy_buffers_size  256k;
```

## Architecture Notes

### How OAuth 2.0 / OpenID Connect Works (simplified)

```
1. User clicks "Sign In" in the app
2. App redirects user to Microsoft login:
   → https://login.microsoftonline.com/{tenant}/oauth2/v2.0/authorize
     ?client_id=...
     &redirect_uri=...        ← THIS must match what's registered
     &response_type=code
     &scope=openid profile email
3. User authenticates at Microsoft
4. Microsoft redirects user back to the app's redirect URI with a code:
   → https://your-app.com/callback?code=abc123
5. App exchanges the code for tokens (server-side, using client_secret)
6. User is logged in
```

The redirect URI is validated at step 2 (authorize) and again at step 5 (token exchange). Both must match exactly.

## ERPNext / Frappe: Social Login Key Setup

ERPNext uses Frappe's **Social Login Key** doctype for SSO. This section covers the exact setup steps and critical pitfalls discovered during the MyProjects deployment.

### Prerequisites

- Azure AD app registration already created (see steps above)
- Client ID and Client Secret ready
- Redirect URI registered in Azure: `https://<domain>/api/method/frappe.integrations.oauth2_logins.login_via_office365`

### Configure via Frappe UI (Recommended)

1. Log into ERPNext as Administrator
2. Navigate to: **Setup → Integrations → Social Login Key**
3. Click **"+ Add Social Login Key"**
4. Select **Social Login Provider**: `Office 365`
5. Fill in:
   - **Provider Name**: `Office 365`
   - **Client ID**: Your Azure AD Application (Client) ID
   - **Client Secret**: Your Azure AD Client Secret Value
   - **Base URL**: `https://login.microsoftonline.com`
   - **Authorize URL**: `https://login.microsoftonline.com/<TENANT_ID>/oauth2/v2.0/authorize`
   - **Access Token URL**: `https://login.microsoftonline.com/<TENANT_ID>/oauth2/v2.0/token`
   - **Redirect URL**: `/api/method/frappe.integrations.oauth2_logins.login_via_office365`
   - **API Endpoint**: `https://graph.microsoft.com/v1.0/me`
   - **API Endpoint Args**: `{"select": "mail,givenName,surname"}`
   - **Auth URL Data**: `{"scope": "openid profile email User.Read", "response_type": "code"}`
   - **Sign Ups**: `Allow` (so new SSO users get auto-created)
   - **Enable Social Login**: checked
6. **Save** — this stores the client_secret properly in Frappe's encrypted `__Auth` table

### Critical: Document Naming

Frappe auto-names the document using `frappe.scrub(provider_name)`. For "Office 365", this produces `office_365`. The callback function `login_via_office365()` **hardcodes** `"office_365"` as the provider key. If the document name doesn't match, the OAuth callback will fail with a `KeyError`.

**Always verify** the document name matches:
```sql
SELECT name FROM `tabSocial Login Key` WHERE name='office_365';
```

### Critical: Client Secret Storage

Frappe stores passwords in the `__Auth` table with encryption. The `client_secret` field in `tabSocial Login Key` should contain `***` — the real value lives in `__Auth`.

**If you set the client_secret via direct SQL** (instead of the Frappe UI), it will be stored in plaintext in the wrong table. Frappe's `get_decrypted_password()` only reads from `__Auth` where `encrypted=1`, so the login button won't appear.

**Diagnosis** — login page shows the `social-logins` div but no button:
```sql
-- Check if secret is in __Auth (where Frappe looks)
SELECT doctype, name, fieldname, encrypted
FROM `__Auth`
WHERE doctype='Social Login Key' AND name='office_365';

-- If empty, the secret is missing or stored wrong
-- Check if it's in plaintext on the doc (wrong place)
SELECT client_secret FROM `tabSocial Login Key` WHERE name='office_365';
-- If this shows the actual secret instead of '***', it needs to be moved
```

**Fix** — use `bench execute` to properly encrypt and store:
```python
# Create a file: /tmp/fix_secret.py
import frappe
from frappe.utils.password import set_encrypted_password

def run():
    # If secret is in plaintext on the doc:
    secret = frappe.db.get_value("Social Login Key", "office_365", "client_secret")
    if secret and secret != '***':
        # Delete any existing __Auth entry
        frappe.db.sql(
            "DELETE FROM `__Auth` WHERE doctype='Social Login Key' "
            "AND name='office_365' AND fieldname='client_secret'"
        )
        # Store with proper encryption
        set_encrypted_password("Social Login Key", "office_365", secret, "client_secret")
        # Clear plaintext from doc
        frappe.db.set_value("Social Login Key", "office_365", "client_secret", "***")
        frappe.db.commit()
        print(f"Fixed: secret ({len(secret)} chars) moved to __Auth")
    else:
        print("Secret already in __Auth or missing")
    return "done"
```
```bash
# Copy to frappe apps dir and execute
docker cp /tmp/fix_secret.py frappe-backend:/home/frappe/frappe-bench/apps/frappe/frappe/fix_secret.py
docker exec frappe-backend bench --site <site-name> execute frappe.fix_secret.run
# Clean up
docker exec frappe-backend rm /home/frappe/frappe-bench/apps/frappe/frappe/fix_secret.py
```

### Critical: Sign Ups Must Be Allowed

If the Social Login Key's `sign_ups` field is empty, Frappe falls back to the global Website Settings `disable_signup` flag. If signup is disabled globally (common in production ERPNext), SSO users who don't already have an ERPNext account will see **"Signup is Disabled"**.

**Fix**:
```sql
UPDATE `tabSocial Login Key` SET sign_ups='Allow' WHERE name='office_365';
```
Or set it in the Frappe UI: Social Login Key → Office 365 → **Sign Ups** = `Allow`.

### Verification Checklist

After setup, verify all of these:

```sql
-- 1. Document name is correct (must be office_365, not "Office 365")

FROM `tabSocial Login Key` WHERE name='office_365';
-- Expected: office_365 | Office 365 | Office 365 | 1 | Allow

-- 2. Client secret is in __Auth with encryption

-- Expected: Social Login Key | office_365 | client_secret | 1

-- 3. Client secret is NOT in plaintext on the doc

```

Then clear cache and test:
```bash

```

Visit the login page — you should see a **"Login with Office 365"** button.

### Auto-Login (Skip the Login Page)

ERPNext doesn't natively support auto-redirect to SSO. Users must click the "Login with Office 365" button. To implement auto-login, you would need to customize the Frappe login page template or add client-side JavaScript to auto-click the SSO button.

### Why "Web" Platform, Not "SPA"

- **Web**: App has a server that exchanges the authorization code using a client_secret. The redirect URI receives a POST with the code. This is what NextAuth, ERPNext, Django, Laravel, etc. use.
- **SPA**: App runs entirely in the browser with no server-side code exchange. The redirect URI receives the token directly in the URL fragment. This is for pure client-side React/Vue/Angular apps.

Using the wrong platform type causes AADSTS900561 ("endpoint only accepts POST requests").
